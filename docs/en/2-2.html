Topic 01

async / await Basics

Swift Concurrency makes asynchronous code read like synchronous code
Swift
// ❌ Callback approach — nesting gets deep (Callback Hell)
func fetchUser(completion: @escaping (User?) -> Void) {
    URLSession.shared.dataTask(with: url) { data, _, _ in
        guard let data = data else { completion(nil); return }
        completion(try? JSONDecoder().decode(User.self, from: data))
    }.resume()
}

// ✅ async/await approach — reads top to bottom
func fetchUser() async throws -> User {
    let (data, _) = try await URLSession.shared.data(from: url)
    return try JSONDecoder().decode(User.self, from: data)
}

// await: "Wait here, continue when the result arrives"
// async: "This function can only be called in an async context"
// throws: "This function can throw an error, handle it with try"

// How to call it
Task {
    do {
        let user = try await fetchUser()
        print(user.name)
    } catch {
        print("Error: \(error)")
    }
}
Think About It
  • When you hit await, does the current thread stop, or does it go handle other work?
  • When calling an async throws function, why is the order try await and not the other way around?
  • What is the difference between SwiftUI View's .task { } and onAppear { Task { } }?
What await Means
Suspends the current execution and waits for the result. It does not block the thread — other work can use that thread
Task { }
A container to start async code from a synchronous context. Supports cancellation and priority settings
.task modifier
Starts a Task when the View appears, automatically cancels when it disappears. An improved version of onAppear + Task
Topic 02

URLSession — HTTP Requests

The basic pattern for fetching data from an API with GET requests
Swift
// API response model
struct Post: Codable, Identifiable {
    let id: Int
    let title: String
    let body: String
    let userId: Int
}

// Network function
func fetchPosts() async throws -> [Post] {
    guard let url = URL(
        string: "https://jsonplaceholder.typicode.com/posts"
    ) else {
        throw URLError(.badURL)
    }

    // data(from:) supports async/await on iOS 15+
    let (data, response) = try await URLSession.shared
        .data(from: url)

    // Check HTTP status code
    guard let httpResponse = response as? HTTPURLResponse,
          httpResponse.statusCode == 200 else {
        throw URLError(.badServerResponse)
    }

    let decoder = JSONDecoder()
    // Auto-convert snake_case API keys
    // decoder.keyDecodingStrategy = .convertFromSnakeCase
    return try decoder.decode([Post].self, from: data)
}

// POST request (sending data)
func createPost(_ post: Post) async throws -> Post {
    guard let url = URL(
        string: "https://jsonplaceholder.typicode.com/posts"
    ) else { throw URLError(.badURL) }

    var request = URLRequest(url: url)
    request.httpMethod = "POST"
    request.setValue("application/json",
        forHTTPHeaderField: "Content-Type")
    request.httpBody = try JSONEncoder().encode(post)

    let (data, _) = try await URLSession.shared
        .data(for: request)
    return try JSONDecoder().decode(Post.self, from: data)
}
Think About It
  • What do HTTP status codes 200, 404, and 500 each mean?
  • What is the difference between URLSession.shared and a custom URLSession? When should you create your own?
  • Where and how should you include an API authentication token? (Header vs Query parameter)
URLSession.shared.data(from:)
The simplest form of a GET request. Returns a (Data, URLResponse) tuple
URLRequest
A request object containing HTTP Method, Header, and Body. Used for POST/PUT/PATCH/DELETE
keyDecodingStrategy
.convertFromSnakeCase: auto-converts user_name → userName. Maps without CodingKeys
Topic 03

Loading / Success / Error State Handling

How to represent the 3 states of a network request in your View
Swift
// Represent state with an enum
enum LoadingState {
    case idle
    case loading
    case success(T)
    case failure(Error)
}

struct PostListView: View {
    @State private var state: LoadingState<[Post]> = .idle

    var body: some View {
        Group {
            switch state {
            case .idle:
                Color.clear.onAppear { load() }
            case .loading:
                ProgressView("Loading...")
            case .success(let posts):
                List(posts) { post in
                    VStack(alignment: .leading, spacing: 4) {
                        Text(post.title)
                            .font(.headline)
                        Text(post.body)
                            .font(.caption)
                            .foregroundStyle(.secondary)
                            .lineLimit(2)
                    }
                }
            case .failure(let error):
                VStack(spacing: 16) {
                    Image(systemName: "wifi.slash")
                        .font(.system(size: 40))
                        .foregroundStyle(.secondary)
                    Text(error.localizedDescription)
                        .multilineTextAlignment(.center)
                    Button("Retry") { load() }
                        .buttonStyle(.borderedProminent)
                }
                .padding()
            }
        }
        .navigationTitle("Posts")
    }

    func load() {
        state = .loading
        Task {
            do {
                let posts = try await fetchPosts()
                state = .success(posts)
            } catch {
                state = .failure(error)
            }
        }
    }
}
Think About It
  • What are the advantages of representing LoadingState as an enum compared to splitting it into @State var isLoading: Bool + @State var error: Error??
  • When loading data with .task { }, what happens if the View disappears?
  • How do you call multiple APIs simultaneously on the same screen? (async let)
enum State Modeling
Explicitly represents possible states. The compiler catches missing state handling (switch exhaustiveness check)
async let — Parallel Requests
async let a = fetchA(); async let b = fetchB(); let (ra, rb) = try await (a, b) — runs both requests concurrently
ProgressView
Loading indicator component. ProgressView("text") for spinner, ProgressView(value: 0.5) for progress bar
Topic 04

@MainActor — UI Updates on the Main Thread

How to safely reflect data received in the background onto the UI
Swift
// When you add @MainActor to a ViewModel,
// all property changes run on the main thread
@MainActor
class NewsViewModel: ObservableObject {
    @Published var articles: [Article] = []
    @Published var isLoading = false
    @Published var errorMessage: String?

    func fetchArticles() async {
        isLoading = true
        errorMessage = nil

        do {
            // Network runs in the background
            articles = try await NewsService.fetch()
        } catch {
            // Safe for UI updates because of @MainActor
            errorMessage = error.localizedDescription
        }

        isLoading = false
    }
}

struct NewsView: View {
    @StateObject private var vm = NewsViewModel()

    var body: some View {
        NavigationStack {
            Group {
                if vm.isLoading {
                    ProgressView()
                } else {
                    List(vm.articles) { article in
                        Text(article.title)
                    }
                }
            }
            .navigationTitle("News")
        }
        .task { await vm.fetchArticles() }
    }
}
Think About It
  • Why must UI updates happen on the main thread?
  • What warning/error occurs if you change a @Published property from a background thread without @MainActor?
  • Between @StateObject and @ObservedObject, which should you use in the View that creates the ViewModel?
@MainActor
Guarantees that a class or function always runs on the main thread. Required for UIKit/SwiftUI updates
Actor
A Swift Concurrency primitive that prevents data race conditions. MainActor is a special global actor
Topic 05

Network Error Handling

How to define predictable errors as types and inform users
Swift
// Custom error type
enum NetworkError: LocalizedError {
    case invalidURL
    case badResponse(statusCode: Int)
    case decodingFailed
    case noInternet

    var errorDescription: String? {
        switch self {
        case .invalidURL:
            return "Invalid URL."
        case .badResponse(let code):
            return "Server error (\(code)). Please try again later."
        case .decodingFailed:
            return "Unable to read the data."
        case .noInternet:
            return "Please check your internet connection."
        }
    }
}

// Using the error type
func fetchData(from urlString: String) async throws -> Data {
    guard let url = URL(string: urlString) else {
        throw NetworkError.invalidURL
    }

    let (data, response) = try await URLSession.shared
        .data(from: url)

    guard let http = response as? HTTPURLResponse else {
        throw NetworkError.badResponse(statusCode: 0)
    }

    guard (200...299).contains(http.statusCode) else {
        throw NetworkError.badResponse(
            statusCode: http.statusCode)
    }

    return data
}

// Branched error handling
do {
    let data = try await fetchData(from: "https://...")
    let posts = try JSONDecoder().decode([Post].self, from: data)
} catch let error as NetworkError {
    print(error.errorDescription ?? "")
} catch is DecodingError {
    print("JSON parsing failed")
} catch {
    print("Unknown error: \(error)")
}
Think About It
  • What are the benefits of adopting the LocalizedError protocol?
  • What is the execution order of catch let error as NetworkError vs catch? Should more specific catches come first?
  • How would you implement automatic retry logic on network failure?
LocalizedError
Implement errorDescription so error.localizedDescription returns a user-friendly message
Layered Error Catching
Catch specific error types first. The final catch error is a fallback that handles all remaining errors
Topic 06

Practical Network Layer Pattern

Building a reusable API client
Swift
// Reusable generic network function
struct APIClient {
    static let shared = APIClient()
    private let baseURL = "https://jsonplaceholder.typicode.com"
    private let decoder: JSONDecoder = {
        let d = JSONDecoder()
        d.keyDecodingStrategy = .convertFromSnakeCase
        return d
    }()

    func fetch(_ path: String) async throws -> T {
        guard let url = URL(string: baseURL + path) else {
            throw NetworkError.invalidURL
        }
        let (data, response) = try await URLSession.shared
            .data(from: url)
        guard let http = response as? HTTPURLResponse,
              (200...299).contains(http.statusCode) else {
            throw NetworkError.badResponse(statusCode: 0)
        }
        do {
            return try decoder.decode(T.self, from: data)
        } catch {
            throw NetworkError.decodingFailed
        }
    }
}

// Usage — clean with type inference
let posts: [Post] = try await APIClient.shared.fetch("/posts")
let user: User = try await APIClient.shared.fetch("/users/1")
Think About It
  • In fetch<T: Decodable>, how is T determined at the call site?
  • What are the pros of the singleton APIClient.shared and its downsides in testing?
  • How would you automatically add a Bearer token to all requests for authenticated APIs?
Generic fetch Function
The return type is determined at the call site. URL, headers, and error handling are centralized, eliminating duplication
Singleton Pattern
Shares a single instance across the entire app. Configuration defined once. Downside: difficult to mock in tests
