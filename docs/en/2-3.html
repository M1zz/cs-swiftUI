Topic 01

Protocol — A Contract for Types

An interface that enforces implementation — any type that adopts it must fulfill its requirements
Swift
// Protocol declaration — "Implement this to be Drawable"
protocol Drawable {
    func draw() -> String
}

protocol Describable {
    var description: String { get }
}

// Adopting multiple Protocols
struct Circle: Drawable, Describable {
    let radius: Double

    func draw() -> String { "◯ r=\(radius)" }
    var description: String { "Circle(radius: \(radius))" }
}

struct Rectangle: Drawable, Describable {
    let width: Double
    let height: Double

    func draw() -> String { "▭ \(width)×\(height)" }
    var description: String { "Rect(\(width)×\(height))" }
}

// Polymorphism via Protocol type
let shapes: [any Drawable] = [
    Circle(radius: 5),
    Rectangle(width: 10, height: 3),
    Circle(radius: 2)
]

// Call draw() without knowing the concrete type
shapes.forEach { print($0.draw()) }
// ◯ r=5.0
// ▭ 10.0×3.0
// ◯ r=2.0

// Protocol with default implementation (Extension)
protocol Shape: Drawable {
    var area: Double { get }
    func describe() -> String
}

extension Shape {
    // Default implementation — adopting types work without override
    func describe() -> String {
        "The area of this shape is \(area)."
    }
}
Think About It
  • What does protocol Shape: Drawable mean? (protocol inheriting from another protocol)
  • What is the difference between any Drawable and some Drawable?
  • What is the difference between { get } and { get set } in Protocol property requirements?
Protocol = Contract
Adopting types must implement all requirements. Verified at compile time, so no runtime errors
Protocol Polymorphism
Group different types under the same Protocol type in an array and call the same methods
Protocol + Extension
Provide default implementations in an Extension so adopting types only override when needed. Eliminates code duplication
Topic 02

Extension — Adding Functionality to Existing Types

Add new features to existing types like String, Int, and Array without access to their source code
Swift
// String Extension
extension String {
    var isEmail: Bool {
        contains("@") && contains(".")
    }

    var trimmed: String {
        trimmingCharacters(in: .whitespaces)
    }

    func repeated(_ times: Int) -> String {
        String(repeating: self, count: times)
    }
}

"hello@example.com".isEmail   // true
"  hello  ".trimmed            // "hello"
"ha".repeated(3)               // "hahaha"

// Int Extension
extension Int {
    var isEven: Bool { self % 2 == 0 }
    var isOdd: Bool  { !isEven }

    // Clamp within range
    func clamped(to range: ClosedRange) -> Int {
        min(max(self, range.lowerBound), range.upperBound)
    }
}

7.isOdd                    // true
150.clamped(to: 0...100)   // 100

// Array Extension
extension Array where Element: Comparable {
    var sortedUnique: [Element] {
        Array(Set(self)).sorted()
    }
}

[3, 1, 2, 1, 3].sortedUnique  // [1, 2, 3]

// Collection Extension — safe subscript
extension Collection {
    subscript(safe index: Index) -> Element? {
        indices.contains(index) ? self[index] : nil
    }
}

let arr = [1, 2, 3]
arr[safe: 5]  // nil (no crash)
Think About It
  • Why can't you add stored properties via Extension?
  • When is a conditional Extension like where Element: Comparable useful?
  • What should you be careful about when adding Extensions to Apple's Foundation types (String, Array)?
Extension Limitations
Cannot add stored properties. Only computed properties, methods, initializers, and protocol conformances
Conditional Extension
A where clause activates the Extension only when specific type conditions are met. Preserves type safety
Code Organization Tool
Split the same type into multiple Extensions by feature for better readability. Works well with // MARK: - Protocol
Topic 03

Generic — Type-Independent Code

No need to write separate versions for Int, String, and Double
Swift
// Generic stack data structure
struct Stack {
    private var storage: [Element] = []

    mutating func push(_ element: Element) {
        storage.append(element)
    }

    mutating func pop() -> Element? {
        storage.popLast()
    }

    var top: Element? { storage.last }
    var isEmpty: Bool { storage.isEmpty }
    var count: Int { storage.count }
}

var intStack = Stack()
intStack.push(1); intStack.push(2); intStack.push(3)
intStack.pop()  // 3

var strStack = Stack()
strStack.push("Swift")
strStack.push("Generic")

// Generic function
func swapValues(_ a: inout T, _ b: inout T) {
    let temp = a
    a = b
    b = temp
}

// Generic + where (type constraint)
func findMin(_ array: [T]) -> T? {
    guard !array.isEmpty else { return nil }
    return array.min()
}

func findMax(_ array: [T]) -> T? {
    array.max()
}

findMin([3, 1, 4, 1, 5])  // Optional(1)
findMin(["banana", "apple", "cherry"])  // Optional("apple")

// Multiple type constraints
func process(
    _ items: [T]
) -> [T] {
    Array(Set(items)).sorted()
}

// Generic typealias
typealias Pair = (T, T)
let point: Pair = (3.0, 4.0)
Think About It
  • Why can't you use the < operator without T: Comparable?
  • Swift's Array and Optional are also Generic types. What might their internal implementation look like?
  • some View is an opaque type. How does it differ from Generic's T?
Type Parameter <T>
A placeholder for an actual type. The compiler determines the type at the call site. Achieves both type safety and reusability
Type Constraint
T: Protocol or T: Class specifies capabilities T must have. Enables use of constrained methods/operators
where Clause
Expresses complex type relationships. Can also express associated type relationships like where T == U.Element
Topic 04

throws · try · catch — Error Handling

Express exceptional situations as types and force callers to handle them explicitly
Swift
// Custom error type
enum AppError: LocalizedError {
    case invalidInput(String)
    case notFound(id: Int)
    case permissionDenied
    case unknown(Error)

    var errorDescription: String? {
        switch self {
        case .invalidInput(let msg):
            return "Invalid input: \(msg)"
        case .notFound(let id):
            return "ID \(id) not found"
        case .permissionDenied:
            return "Permission denied"
        case .unknown(let e):
            return "Unknown error: \(e)"
        }
    }
}

// throws function
func validateAge(_ age: Int) throws -> String {
    guard age >= 0 else {
        throw AppError.invalidInput("Age must be 0 or greater")
    }
    guard age <= 150 else {
        throw AppError.invalidInput("Invalid age")
    }
    return age >= 18 ? "Adult" : "Minor"
}

// do-catch error handling
do {
    let result = try validateAge(-5)
    print(result)
} catch AppError.invalidInput(let msg) {
    print("Input error:", msg)   // Input error: Age must be 0 or greater
} catch AppError.permissionDenied {
    print("No permission")
} catch {
    print("Other error:", error)
}

// try? — returns nil on failure (ignores error)
let result1 = try? validateAge(25)    // Optional("Adult")
let result2 = try? validateAge(-1)    // nil

// try! — crashes on failure (use only when certain)
let result3 = try! validateAge(30)    // "Adult"

// Propagating errors upward (rethrows)
func transform(
    _ value: T,
    using block: (T) throws -> T
) rethrows -> T {
    try block(value)
}
Think About It
  • Using try? discards error information. In what situations is this tradeoff acceptable?
  • What is rethrows and why is it needed?
  • How does Swift's error handling differ from Objective-C's NSError approach?
throws = Explicit Error Declaration
Declares in the function signature that errors may be thrown. Callers must handle them, preventing missed error handling
try / try? / try!
try: must use do-catch | try?: converts to Optional | try!: forced (crashes on failure)
LocalizedError
Implement errorDescription to provide user-friendly messages. Can be used directly in Alert
Topic 05

Result<Success, Failure> — Error as a Value

Represent success or failure as a single type — especially useful in async contexts or callbacks
Swift
// Basic Result type usage
func divide(_ a: Double, by b: Double) -> Result {
    guard b != 0 else {
        return .failure(.invalidInput("Cannot divide by zero"))
    }
    return .success(a / b)
}

// Handle with switch
let result = divide(10, by: 2)
switch result {
case .success(let value):
    print("Result:", value)  // Result: 5.0
case .failure(let error):
    print("Error:", error.errorDescription ?? "")
}

// map / flatMap chaining
let doubled = divide(10, by: 2)
    .map { $0 * 2 }           // success → transform value
    // .mapError { ... }       // failure → transform error

// Convert to throws with get()
do {
    let value = try divide(10, by: 0).get()
    print(value)
} catch {
    print(error)  // Cannot divide by zero
}

// Result vs throws — when to use which?
// throws: synchronous functions, immediate control flow
// Result: passing in callbacks, deferred handling, storing as a value

// Store a Result and handle later
struct CachedResult {
    var value: Result?

    func unwrap() throws -> T {
        guard let result = value else {
            throw AppError.notFound(id: 0)
        }
        return try result.get()
    }
}
Think About It
  • How is Result.get() implemented internally?
  • What does Result<T, Never> mean? What is Never?
  • In an async/await environment, when is it appropriate to return a Result type?
Result = Return Value with Error
Represents success (.success) and failure (.failure) as a single enum. Useful for structuring error handling in callback patterns
map / flatMap
Transforms the success case value. On failure, it propagates as-is. Same concept as Optional's map
Interconversion with throws
result.get() converts to throws, and Result { try fn() } converts a throwing function to Result
Topic 06

enum Associated Values — State with Data

Each case can carry different types of data — the most powerful feature of Swift enums
Swift
// Network state — each case carries different data
enum NetworkState {
    case idle
    case loading(progress: Double)
    case success(data: Data, statusCode: Int)
    case failure(error: Error, retryCount: Int)
}

// Pattern matching + value extraction with switch
func handleState(_ state: NetworkState) {
    switch state {
    case .idle:
        print("Idle")
    case .loading(let progress):
        print("Loading: \(Int(progress * 100))%")
    case .success(let data, let code):
        print("Success (\(code)): \(data.count) bytes")
    case .failure(let error, let retry):
        print("Failed (retry \(retry)): \(error)")
    }
}

handleState(.loading(progress: 0.75))
// Loading: 75%

// if case let — extract a single case
let state = NetworkState.success(
    data: Data(), statusCode: 200
)
if case .success(_, let code) = state, code == 200 {
    print("Request succeeded!")
}

// Practical example — app notification types
enum AppNotification {
    case message(from: String, body: String)
    case friendRequest(userId: Int, name: String)
    case systemAlert(title: String, priority: Int)
    case promotion(discount: Double, expiry: Date)
}

func notificationTitle(_ n: AppNotification) -> String {
    switch n {
    case .message(let from, _):      return "Message from \(from)"
    case .friendRequest(_, let name): return "\(name) sent a friend request"
    case .systemAlert(let title, _): return title
    case .promotion(let d, _):       return "\(Int(d*100))% off!"
    }
}
Think About It
  • What is the difference between an enum with associated values and a struct? When do you choose enum?
  • What is the syntax difference between ignoring an associated value with _ and binding it?
  • Swift's Optional<T> is actually an enum composed of .none and .some(T). Explain Optional's behavior from this perspective
Associated Values
Each case can carry values of different types and counts. Represents state + data as a single type
Pattern Matching
Extract case and data simultaneously with switch, if case, guard case. Nested patterns are also possible
Exhaustive switch
Compile error if not all cases are handled. When a new case is added, the compiler catches any missing handling
