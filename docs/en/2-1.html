Topic 01

Codable — Swift ↔ JSON

How to convert data to JSON and restore it back
Swift
// Codable = Encodable + Decodable
struct User: Codable {
    let id: Int
    let name: String
    let email: String
}

// Swift → JSON (Encoding)
let user = User(id: 1, name: "Hyunho", email: "test@test.com")
let encoder = JSONEncoder()
encoder.outputFormatting = .prettyPrinted
if let data = try? encoder.encode(user),
   let json = String(data: data, encoding: .utf8) {
    print(json)
    // { "id": 1, "name": "Hyunho", "email": "test@test.com" }
}

// JSON → Swift (Decoding)
let jsonString = """
{ "id": 1, "name": "Hyunho", "email": "test@test.com" }
"""
let decoder = JSONDecoder()
if let data = jsonString.data(using: .utf8),
   let decoded = try? decoder.decode(User.self, from: data) {
    print(decoded.name) // Hyunho
}
Think About It
  • If the server's JSON key is user_name but you want your Swift property to be userName, how do you handle that? (CodingKeys)
  • What is the difference between try? and try!? Which is safer in a real app?
  • How do you represent nested JSON structures (objects within objects) with Codable?
Codable Protocol
A type alias for Encodable + Decodable. If all properties of a struct are Codable, the implementation is auto-synthesized
JSONEncoder / JSONDecoder
Provided by Swift standard library. Use keyDecodingStrategy for automatic snake_case ↔ camelCase conversion
CodingKeys enum
Maps JSON keys to Swift property names when they differ. case userName = "user_name"
Topic 02

UserDefaults — Simple Settings Storage

The quickest way to persist small data even after the app is closed
Swift
// Saving/loading basic types
UserDefaults.standard.set(true, forKey: "isDarkMode")
UserDefaults.standard.set(42, forKey: "launchCount")
UserDefaults.standard.set("English", forKey: "language")

let isDark = UserDefaults.standard.bool(forKey: "isDarkMode")
let count = UserDefaults.standard.integer(forKey: "launchCount")

// Saving a Codable struct (convert to JSON, then store as Data)
struct AppSettings: Codable {
    var theme: String
    var fontSize: Int
}

let settings = AppSettings(theme: "dark", fontSize: 16)
if let data = try? JSONEncoder().encode(settings) {
    UserDefaults.standard.set(data, forKey: "appSettings")
}

// Loading
if let data = UserDefaults.standard.data(forKey: "appSettings"),
   let saved = try? JSONDecoder().decode(
       AppSettings.self, from: data) {
    print(saved.theme) // dark
}

// @AppStorage — SwiftUI-native UserDefaults
struct ContentView: View {
    @AppStorage("isDarkMode") var isDarkMode = false

    var body: some View {
        Toggle("Dark Mode", isOn: $isDarkMode)
    }
}
Think About It
  • What kind of data is UserDefaults suitable for, and what is it not? (user tokens, cart items, payment info?)
  • What is the biggest difference between @AppStorage and @State?
  • How can you reset UserDefaults values without deleting the app?
When to Use UserDefaults
App settings, user preferences, last launch date — small amounts of simple data. Not suitable for large data or sensitive information
@AppStorage
Use UserDefaults as a binding like @State in SwiftUI. When the value changes, the View automatically re-renders
Topic 03

@Model — SwiftData Data Model

A modern data persistence solution replacing Core Data on iOS 17+
Swift
import SwiftData

// Declare a persistent model with the @Model macro
@Model
class TodoItem {
    var title: String
    var isCompleted: Bool
    var createdAt: Date

    init(title: String) {
        self.title = title
        self.isCompleted = false
        self.createdAt = Date()
    }
}

// Connect ModelContainer at the app entry point
@main
struct MyApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
        }
        .modelContainer(for: TodoItem.self)
    }
}
Think About It
  • Why must SwiftData's @Model be applied to a class rather than a struct?
  • Why can all child Views use it when you attach .modelContainer(for:) to the top-level App?
  • How do you configure .modelContainer to use multiple Model types together?
@Model Macro
Attach to a class and SwiftData automatically handles persistence, change tracking, and relationship management. Requires iOS 17+
ModelContainer
Manages the database file. Register multiple models with .modelContainer(for: [A.self, B.self])
ModelContext
The entry point for CRUD operations. Access from a View via @Environment(\.modelContext). insert / delete / save
Topic 04

@Query — Querying Data

How to read, filter, and sort data in SwiftData
Swift
struct TodoListView: View {
    // @Query: Automatically fetches data from the DB
    // and updates the View when data changes
    @Query(sort: \TodoItem.createdAt, order: .reverse)
    var todos: [TodoItem]

    // Adding filter conditions
    @Query(filter: #Predicate { !$0.isCompleted },
           sort: \TodoItem.createdAt)
    var activeTodos: [TodoItem]

    var body: some View {
        List(todos) { todo in
            Text(todo.title)
        }
    }
}

// CRUD with ModelContext
struct AddTodoView: View {
    @Environment(\.modelContext) private var context
    @State private var title = ""

    var body: some View {
        HStack {
            TextField("To-do", text: $title)
            Button("Add") {
                let item = TodoItem(title: title)
                context.insert(item)  // Create
                title = ""
            }
        }
    }
}

// Delete
func delete(_ item: TodoItem) {
    context.delete(item)
}
Think About It
  • How does an array fetched with @Query differ from @State? Can you modify it directly?
  • After context.insert(), do you need to call try context.save() or not?
  • How do you combine multiple conditions with AND / OR in #Predicate?
@Query Property Wrapper
A live query that observes the DB. Automatically refreshes the View when data changes. Supports sort, filter, and order parameters
#Predicate Macro
Type-safe filter conditions. #Predicate<TodoItem> { $0.isCompleted == false }
Auto-Save
SwiftData automatically saves changes. Explicit save() is optional, but recommended at critical points
Topic 05

Relationships — Connected Data

How to express one-to-many and many-to-many relationships between models
Swift
@Model
class Category {
    var name: String
    // One-to-many: one category has many todos
    @Relationship(deleteRule: .cascade)
    var todos: [TodoItem] = []

    init(name: String) { self.name = name }
}

@Model
class TodoItem {
    var title: String
    var isCompleted: Bool
    // Inverse relationship
    var category: Category?

    init(title: String) {
        self.title = title
        self.isCompleted = false
    }
}

// Setting up relationships
let category = Category(name: "Work")
let todo = TodoItem(title: "Prepare for meeting")
todo.category = category
context.insert(category)

// deleteRule: .cascade → deleting a category also deletes its todos
Think About It
  • What is the difference between deleteRule: .cascade and .nullify? When is each appropriate?
  • When setting up bidirectional relationships, do you need to add @Relationship on both models?
  • Do you need to save the Category first, or the TodoItem first?
@Relationship
Controls cascading deletes via deleteRule. .cascade / .nullify / .deny / .noAction
Inverse Relationships
When both models reference each other, SwiftData automatically keeps them in sync. It works even if only one side is set up
Topic 06

Preview and Test Containers

How to use SwiftData in SwiftUI Previews
Swift
// In-memory container for previews
#Preview {
    let config = ModelConfiguration(isStoredInMemoryOnly: true)
    let container = try! ModelContainer(
        for: TodoItem.self,
        configurations: config
    )

    // Insert sample data
    let context = container.mainContext
    context.insert(TodoItem(title: "Study SwiftData"))
    context.insert(TodoItem(title: "Deploy app"))

    return TodoListView()
        .modelContainer(container)
}

// Production app — persistent storage
@main
struct MyApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
        }
        .modelContainer(for: TodoItem.self)
    }
}
Think About It
  • A container created with isStoredInMemoryOnly: true loses its data when the app restarts. In what situations is this useful?
  • What happens to existing user data when you add a new property to an @Model class? (Schema Migration)
  • How do you set up SwiftData for use in Unit Tests?
In-Memory Container
A temporary DB that doesn't write to disk, for previews and tests. isStoredInMemoryOnly: true
Schema Migration
Migration to preserve existing data when the model structure changes. Uses VersionedSchema and SchemaMigrationPlan
